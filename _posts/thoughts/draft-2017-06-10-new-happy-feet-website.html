A little talk about Jekyll

The only reason I landed up looking into Jekyll (a static website generator) was because I finally started using Github to backup and version control my code. The more I learned about Jekyll, the more iteresting it sounded, and there was no going back. Between the multiple cups of coffee/tea that were consumed, and the multiple hours I landed up ignoring the missus, the migration of this website too place.

For those of you who are interested in reading and learning about Jekyll, the documentation on the official Jekyll website is a great place to start, along with these really simple videos on YouTube, by Thomas Bradley.

Issues with WordPress

Although I would still recommend WordPress to folks wanting to have a website/blog running in a few minutes, it wasn't doing the job for us. Even though we did not publish a lot of posts over the past couple of years, our website had grown to having around 150 posts over these few years. Which meant it was starting to blow up even though most of our photographs were hosted on my Flickr account. Now, no doubt some of the post header images are hosted on the website itself, the size of the WordPress website files (excluding the images hosted on Flickr and the MySql database), after being zipped up, amounted to 520 MB. Now compare that to the Jekyll files, (after zipped) amounting to just a little over 1MB. That's a huge amount of data saved.

Another problem was that WordPress was painstakingly slow, even after removing a ton of plugins, optimizing the code, serving images from a different host, and a lot of caching. What did not help was that the website was also hosted on a shared host. But even after accounting for that, the website was really slow. Period.

WordPress was also prone to a lot of spamming and hijacking attacks, which luckily we were able to fence off, but a recent JavaScript code injection was the icing on the cake. Luckily, I had already moved the website by then, but the backup on the shared host was infected, along with a bunch of other websites.

The bottom line was WordPress was not cutting it for us anymore and not serving our workflow. I upload the photos to my Flickr account. The missus anyway writes all her posts on Google Docs, which I paste into the WordPress editor. We then copy the image code from a script that I wrote, and then paste the code in the places required. So, along with a ton of other reasons that I've not bothered to touch, WordPress was not of any great use anymore.

Plain Text Website

Our entire website now is just plain text. All the photographs are hosted on Flickr (besides the logo and the image loading gif). The best part of this is that I can pull the website source code from Github really quickly, even on some painstakingly slow internet connections in India. The other benefit of having the photos on Flickr is that we don't have to fill in the image titles, alt tags and caption in our posts, as these get pulled directly from Flickr, via the Flickr API. This not only leaves our photos well documented on Flickr, but also cuts down on a ton of time while making our posts.

Dynamic image sizes & lazy loading

Inorder to provide a better user experience, we now serve images based on the user's device screen size. No point sending a 2048 pixel wide image if the user's screen is only around 400 pixels wide. Here is where Flickr comes to the rescue once again.

Luckily photographs hosted on Flickr are automatically resized and made available to the user. The various sizes can be explored here: flickr.photos.getSizes API Docs. These sizes along with lazysizes by Alexander Farkas gets the job done for us.
